# C++ 디자인 패턴 - 장단점 정리

---

# 생성 패턴 (Creational)

---

## Singleton

| 구분 | 내용 |
|------|------|
| **장점** | - 인스턴스가 하나임을 보장하여 메모리 낭비 방지 |
|          | - 전역 접근점 제공으로 어디서든 동일 객체 사용 가능 |
|          | - 최초 호출 시에만 생성되므로 lazy initialization 가능 |
| **단점** | - 전역 상태를 만들어 모듈 간 결합도 증가 |
|          | - 멀티스레드 환경에서 동기화 처리 필요 |
|          | - 단위 테스트 시 mock 교체가 어려움 |
| **적합한 상황** | DB 커넥션 풀, 로거, 설정 매니저 등 단일 자원 관리 |

---

## Factory Method

| 구분 | 내용 |
|------|------|
| **장점** | - 객체 생성과 사용을 분리하여 결합도 감소 |
|          | - 새로운 제품 추가 시 기존 코드 수정 없이 확장 가능 (OCP) |
|          | - 서브클래스에서 생성 로직을 자유롭게 변경 가능 |
| **단점** | - 제품마다 Creator 서브클래스가 필요해 클래스 수 증가 |
|          | - 단순한 생성에는 오버엔지니어링이 될 수 있음 |
| **적합한 상황** | 프레임워크에서 사용자 정의 객체 생성을 허용할 때 |

---

## Abstract Factory

| 구분 | 내용 |
|------|------|
| **장점** | - 관련 객체군의 일관성 보장 (Windows면 전부 Windows 스타일) |
|          | - 구체 클래스를 클라이언트에서 분리하여 교체 용이 |
|          | - 제품군 단위로 한꺼번에 교체 가능 |
| **단점** | - 새로운 종류의 제품 추가 시 모든 팩토리 수정 필요 |
|          | - 클래스 계층이 복잡해질 수 있음 |
| **적합한 상황** | 크로스 플랫폼 UI, 테마 시스템 등 제품군 단위 생성 |

---

## Builder

| 구분 | 내용 |
|------|------|
| **장점** | - 복잡한 객체를 단계별로 조립 가능 |
|          | - 같은 생성 과정으로 다양한 표현 생성 가능 |
|          | - 메서드 체이닝으로 가독성 높은 코드 작성 가능 |
| **단점** | - 단순한 객체에는 불필요한 복잡성 추가 |
|          | - Builder 클래스를 별도로 유지보수해야 함 |
| **적합한 상황** | SQL 쿼리 빌더, 복잡한 설정 객체, UI 레이아웃 구성 |

---

## Prototype

| 구분 | 내용 |
|------|------|
| **장점** | - 복잡한 초기화 과정을 거친 객체를 빠르게 복제 가능 |
|          | - 런타임에 동적으로 객체 타입 결정 가능 |
|          | - 서브클래싱 없이 객체 생성 가능 |
| **단점** | - 깊은 복사(deep copy) 구현이 까다로울 수 있음 |
|          | - 순환 참조가 있는 객체 복제 시 주의 필요 |
| **적합한 상황** | 게임 오브젝트 복제, 문서 템플릿, 설정 프리셋 |

---

# 구조 패턴 (Structural)

---

## Adapter

| 구분 | 내용 |
|------|------|
| **장점** | - 호환되지 않는 인터페이스를 연결하여 재사용성 확보 |
|          | - 기존 코드 수정 없이 서드파티 라이브러리 통합 가능 |
|          | - 단일 책임 원칙 준수 (변환 로직 분리) |
| **단점** | - 어댑터 클래스가 추가되어 전체 복잡도 증가 |
|          | - 과도한 사용 시 코드 흐름 파악이 어려워짐 |
| **적합한 상황** | 레거시 코드 통합, 서드파티 라이브러리 래핑 |

---

## Bridge

| 구분 | 내용 |
|------|------|
| **장점** | - 추상화와 구현을 독립적으로 확장 가능 |
|          | - 런타임에 구현체 교체 가능 |
|          | - 클래스 폭발(조합 수 증가) 문제 방지 |
| **단점** | - 초기 설계 복잡도가 높음 |
|          | - 간단한 구조에는 오버엔지니어링 |
| **적합한 상황** | 렌더링 엔진 교체, 드라이버 추상화, 플랫폼 독립 코드 |

---

## Composite

| 구분 | 내용 |
|------|------|
| **장점** | - 개별 객체와 복합 객체를 동일하게 처리 가능 |
|          | - 트리 구조를 자연스럽게 표현 |
|          | - 새로운 구성 요소 추가가 용이 |
| **단점** | - 지나치게 일반적인 설계가 될 수 있음 |
|          | - 특정 구성 요소만 제한하기 어려움 |
| **적합한 상황** | 파일 시스템, UI 컴포넌트 트리, 조직도 |

---

## Decorator

| 구분 | 내용 |
|------|------|
| **장점** | - 런타임에 동적으로 기능 추가/제거 가능 |
|          | - 상속보다 유연한 기능 확장 |
|          | - 데코레이터 조합으로 다양한 변형 생성 가능 |
| **단점** | - 데코레이터가 많아지면 디버깅이 어려움 |
|          | - 객체 생성 코드가 복잡해질 수 있음 |
| **적합한 상황** | I/O 스트림 래핑, 커피 옵션 조합, 미들웨어 체인 |

---

## Facade

| 구분 | 내용 |
|------|------|
| **장점** | - 복잡한 서브시스템을 단순한 인터페이스로 제공 |
|          | - 클라이언트와 서브시스템 간 결합도 감소 |
|          | - 진입 장벽을 낮춰 사용성 향상 |
| **단점** | - 퍼사드가 모든 기능을 감싸면 God Object가 될 수 있음 |
|          | - 서브시스템의 세밀한 제어가 제한될 수 있음 |
| **적합한 상황** | 미디어 플레이어, 컴파일러 프론트엔드, SDK 래퍼 |

---

## Flyweight

| 구분 | 내용 |
|------|------|
| **장점** | - 공유를 통해 메모리 사용량 대폭 절감 |
|          | - 대량의 유사 객체를 효율적으로 관리 |
| **단점** | - 공유 상태와 고유 상태 분리 설계가 필요 |
|          | - 코드 복잡도 증가, 스레드 안전성 고려 필요 |
| **적합한 상황** | 게임 맵 타일, 텍스트 에디터 글자 객체, 입자 시스템 |

---

## Proxy

| 구분 | 내용 |
|------|------|
| **장점** | - 실제 객체 접근 전 추가 로직 삽입 가능 (캐싱, 권한, 로깅) |
|          | - 무거운 객체의 lazy initialization 가능 |
|          | - 클라이언트 코드 변경 없이 제어 로직 추가 |
| **단점** | - 프록시 계층이 추가되어 응답 지연 가능 |
|          | - 코드 복잡도 증가 |
| **적합한 상황** | 이미지 lazy loading, 원격 프록시(RPC), 접근 제어 |

---

# 행동 패턴 (Behavioral)

---

## Chain of Responsibility

| 구분 | 내용 |
|------|------|
| **장점** | - 요청 발신자와 수신자 간 결합도 제거 |
|          | - 핸들러 추가/제거/순서 변경이 유연 |
|          | - 각 핸들러가 단일 책임만 가짐 |
| **단점** | - 요청이 체인 끝까지 처리되지 않을 수 있음 |
|          | - 디버깅 시 어떤 핸들러가 처리했는지 추적 어려움 |
| **적합한 상황** | 고객지원 에스컬레이션, 이벤트 버블링, 미들웨어 |

---

## Command

| 구분 | 내용 |
|------|------|
| **장점** | - 실행 취소(Undo) / 재실행(Redo) 구현 용이 |
|          | - 명령을 큐에 넣어 지연 실행, 로깅, 트랜잭션 가능 |
|          | - 요청 발신자와 수신자 완전 분리 |
| **단점** | - 간단한 동작에도 Command 클래스가 필요하여 클래스 수 증가 |
|          | - 복잡한 Undo 로직 시 상태 관리 어려움 |
| **적합한 상황** | 텍스트 에디터 Undo, GUI 버튼 바인딩, 매크로 기록 |

---

## Interpreter

| 구분 | 내용 |
|------|------|
| **장점** | - 문법 규칙을 클래스로 표현하여 확장 용이 |
|          | - 간단한 DSL(도메인 특화 언어) 구현에 적합 |
| **단점** | - 복잡한 문법에는 클래스 수가 폭발적으로 증가 |
|          | - 성능이 좋지 않아 대규모 파싱에는 부적합 |
| **적합한 상황** | 수식 파서, 정규표현식 엔진, 간단한 스크립트 언어 |

---

## Iterator

| 구분 | 내용 |
|------|------|
| **장점** | - 컬렉션 내부 구조를 노출하지 않고 순회 가능 |
|          | - 동일 인터페이스로 다양한 컬렉션 순회 |
|          | - 다양한 순회 방식(정방향, 역방향 등) 제공 가능 |
| **단점** | - 단순 컬렉션에는 오버엔지니어링 |
|          | - 순회 중 컬렉션 수정 시 문제 발생 가능 |
| **적합한 상황** | 커스텀 컬렉션 순회, 트리/그래프 탐색 |

---

## Mediator

| 구분 | 내용 |
|------|------|
| **장점** | - 객체 간 직접 참조를 제거하여 결합도 감소 |
|          | - 상호작용 로직을 한 곳에 집중하여 관리 용이 |
|          | - 개별 컴포넌트 재사용성 향상 |
| **단점** | - 중재자가 비대해지면 God Object가 될 수 있음 |
|          | - 중재자 자체가 단일 장애점이 될 수 있음 |
| **적합한 상황** | 채팅룸, UI 컴포넌트 간 통신, 항공 관제탑 |

---

## Memento

| 구분 | 내용 |
|------|------|
| **장점** | - 캡슐화를 위반하지 않고 객체 상태 저장/복원 가능 |
|          | - Undo 기능 구현이 직관적 |
| **단점** | - 상태가 크면 메모리 사용량 급증 |
|          | - 스냅샷 관리 정책(최대 개수, 만료 등) 필요 |
| **적합한 상황** | 에디터 히스토리, 게임 세이브, 트랜잭션 롤백 |

---

## Observer

| 구분 | 내용 |
|------|------|
| **장점** | - 발행자와 구독자 간 느슨한 결합 |
|          | - 런타임에 구독자 동적 추가/제거 가능 |
|          | - 이벤트 기반 아키텍처 구현에 핵심 |
| **단점** | - 구독자가 많아지면 알림 순서 및 성능 문제 |
|          | - 순환 알림 발생 가능 |
|          | - 메모리 누수 주의 (구독 해제 누락) |
| **적합한 상황** | 주가 알림, UI 이벤트 시스템, 모델-뷰 동기화 |

---

## State

| 구분 | 내용 |
|------|------|
| **장점** | - 상태별 행동을 별도 클래스로 분리하여 가독성 향상 |
|          | - if/switch 분기문 제거로 유지보수 용이 |
|          | - 새로운 상태 추가가 기존 코드에 영향 없음 |
| **단점** | - 상태가 적으면 오버엔지니어링 |
|          | - 상태 전이 로직이 분산되어 전체 흐름 파악 어려울 수 있음 |
| **적합한 상황** | 자판기, TCP 연결 상태, 게임 캐릭터 상태 |

---

## Strategy

| 구분 | 내용 |
|------|------|
| **장점** | - 알고리즘을 런타임에 자유롭게 교체 가능 |
|          | - 알고리즘 추가 시 기존 코드 수정 불필요 (OCP) |
|          | - 조건문 분기를 객체 교체로 대체 |
| **단점** | - 클라이언트가 전략 간 차이를 알아야 선택 가능 |
|          | - 전략 수가 많아지면 클래스 수 증가 |
| **적합한 상황** | 정렬 알고리즘 교체, 결제 수단 선택, 압축 방식 전환 |

---

## Template Method

| 구분 | 내용 |
|------|------|
| **장점** | - 알고리즘 뼈대 재사용, 세부 단계만 오버라이드 |
|          | - 코드 중복 제거 |
|          | - 확장 포인트가 명확 |
| **단점** | - 상속 기반이라 유연성이 Strategy보다 낮음 |
|          | - 상위 클래스 변경 시 하위 클래스 전체에 영향 |
|          | - 단계가 많아지면 이해하기 어려움 |
| **적합한 상황** | 데이터 파이프라인, 테스트 프레임워크, 문서 변환기 |

---

## Visitor

| 구분 | 내용 |
|------|------|
| **장점** | - 객체 구조 변경 없이 새로운 연산 추가 가능 |
|          | - 관련 연산을 하나의 Visitor에 집중 |
|          | - 이중 디스패치(double dispatch)로 타입별 처리 |
| **단점** | - 새로운 요소 타입 추가 시 모든 Visitor 수정 필요 |
|          | - 요소의 내부 상태 접근이 필요하면 캡슐화 위반 |
| **적합한 상황** | AST 처리(컴파일러), 도형 직렬화, 보고서 생성 |
